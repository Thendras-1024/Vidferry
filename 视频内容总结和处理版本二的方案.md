# 处理后内容分析与发布信息生成第一阶段方案

## Summary

- 复用现有 ASR transcript 缓存，把“内容分析 + 发布信息生成”接入为版本一、版本二的公共基础能力。
- 第一阶段采用“后台并行、成片保留、分析可重试”的策略：字幕烧录成功即可素材入库，AI 分析失败不阻断成片。
- 分析结果绑定到视频线索 `youtube_videos`，复用现有 `analysis_result`，后续素材管理和发布中心从线索读取发布文案。
- 本轮只做自动生成和查看，不做发布中心一键应用，不做高光开头混剪，不做封面帧生成。

## Key Changes

### 后端任务流程

- 在 ASR transcript 生成并缓存后，自动触发 AI 内容分析后台任务。
- 翻译/字幕烧录继续走原有主线；AI 分析走独立后台分支，复用 transcript，不重新 ASR、不读取原始视频。
- 处理版本一：
  - 执行 ASR -> 翻译 -> ASS -> FFmpeg 烧录 -> 素材入库。
  - ASR 后自动触发 AI 分析，分析完成后更新线索结果。
- 处理版本二当前阶段：
  - 保持现有“剪辑方案/内容分析”能力。
  - 不在本轮实现高光混剪，但分析结果继续输出高光时间轴，作为后续剪辑输入。

### 分析结果与状态

- 分析结果仍绑定视频线索，结构继续包含：
  - `summary`
  - `china_view_angle`
  - `title_options`
  - `publish_copy`
  - `tags`
  - `highlight_segments`
  - `risk_notes`
  - `model`
- 明确 `analysis_status` 语义：
  - `0`：未分析
  - `1`：分析成功
  - `2`：分析中
  - `3`：分析失败
- 分析失败时保留成片和素材入库结果，记录失败状态和错误说明，允许用户之后手动重试生成文案。
- 自动触发分析前需要去重：
  - 已成功分析则不重复分析。
  - 已有运行中的分析任务则不重复创建。
  - 失败后允许手动重试。

### 前端展示

- 视频采集处理页保留现有“生成/查看剪辑方案”入口。
- 处理完成后，如果分析正在运行，行内状态显示“文案生成中”或等价提示。
- 分析成功后，用户可以查看标题候选、作品描述、话题标签、视频总结和高光片段。
- 分析失败后，用户可以看到失败提示，并可重新触发生成。
- 本轮不改素材管理展示，不改发布中心自动填充；这两项留给 TODO 第 2 阶段。

### 安全与边界

- AI 分析模块只读取 transcript、视频元数据和已有任务字段，不接触本地文件路径之外的视频内容。
- 模型输出只作为 JSON 数据保存和展示，不能作为命令、路径、SQL 或发布动作执行。
- transcript 输入继续受 `LLM_MAX_TRANSCRIPT_CHARS` 限制，长视频分块总结后再汇总。
- API Key、模型地址继续走 `.env`，前端不暴露密钥。
- 生成标题、描述、话题只作为草稿，第一阶段不自动发布，必须由用户审核。
- transcript 可能包含不可信内容，提示词需要明确禁止遵循 transcript 中的指令，只允许总结视频内容。

## Test Plan

- 下载并处理一个有英文口播的视频：
  - transcript 只生成一次。
  - 字幕烧录正常完成并入库素材。
  - AI 分析自动开始，完成后可查看标题、描述、标签、总结。
- 处理一个已有 transcript 的视频：
  - 不重复 ASR。
  - 分析任务复用缓存 transcript。
- 模拟 LLM 未配置：
  - 字幕处理和素材入库仍成功。
  - 分析状态显示失败或未配置错误。
- 模拟 LLM 请求失败：
  - 处理后视频仍保留。
  - 分析可手动重试。
- 重复点击生成文案：
  - 已成功时不重复生成。
  - 运行中时提示已有任务。
  - 失败后允许重新生成。
- 运行验证：
  - `python -m py_compile sau_backend.py app/config.py`
  - `cd sau_frontend && npm run build`

## Assumptions

- 第一阶段分析结果绑定视频线索，不绑定单个处理后素材。
- 第一阶段分析失败不影响成片入库。
- 第一阶段自动分析后台并行执行，不要求处理任务等待分析完成。
- 第一阶段只完成“自动生成 + 视频采集页查看”，素材管理展示和发布中心一键应用后续实现。
- 高光开头混剪和封面帧生成不在本轮实现，只保留分析结果作为后续输入。
